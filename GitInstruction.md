# Git для новичков

![git](git_icon.jpg)

***Git*** является одним из распределенных систем контроля версий. Это системы при которых репозиторий проекта ханится на жестком диске разработчика.

## Основные команды Git:
1. git init  - создает репозиторий
2. git status - отображает текущее состояние git
3. git add - добавляет файл к следующему коммиту
4. git commit -m "message" - создает коммит
5. git log - показывает историю всех коммитов
6. git checkout - позволяет переходить от одного коммита к другому
7. git checkout master - возвращает к актуальному состоянию
8. git diff - позволяет увидеть разницу м/у ауктуальным и каким-то из предыдущих коммитов
 
## Ветвления в Git

![Ветвление в git](git_branch.jpg)

Основная идея ветвления - это отклонение от основного кода и продолжение работы независимо от него. Также это удобно в тестировании отдельного функционала, потому что позволяет работать над новой частью кода, не беспокоясь о полмке чего-то в рабочей версии.

## Основные команды при ветвлении в Git:

1. git branch - вывод всех веток
2. git branch dranch_name - создание ветки с конкретным названием
3. git checkout branch_name - переход на конкретную ветку
4. git checkout maser - переход на основную ветку master
5. git merge branch_name - слияние текущей ветки с основной рабочей веткой master
6. git branch -d branch_name - удаление ветки
7. git log --graph - визуализация всех веток дерева

# Синтаксис языка Markdown

## Создание заголовков разного уровня от H1 до H6

Для создания заголовков используется символ [#], который находится перед ним, при этом есть несколько тонкостей:

* решетки [#] можно ставить как перед заголовками, так и с двух сторн от него (на уровень заголовка влияют тольк те [#], которые находятся перед ним)
* кол-во [#] соответствует уровню заголовка: одна для первого, две для второго и т.д.
* между [#] и текстом ставится пробел

## Выделение текста

### *Курсив*

Для выделения текста курсивом нужно использовать одну [*] перед и после текста, без пробеллов между символами и текстом: *Вот так*

### **Жирный шрифт**

Для выделения текста жиным нужно использовать две [**] по обе стороны от текста: **Вот так**

### ***Жиррный курсив***

Для выделения текста сразу обоими стилями нужно использовать три звездочки по обе стороны от текста: ***Вот так***

## Списки в Markdown

### Нумерованные списки

Для создания нумерованного списка перед пунктами нужно поставить число с точкой. Нумерация в разметке линейная:

1. Первый пункт

2. Второй пункт

3. Третий пункт

### Ненумерованные списки

Для создания немумерованных списков нужно поставить перед каждым пунктом звездочку [*], дефис [-] или плюс [+] и пробел:

* позиция

- позиция

+ позиция

## Оформление цитат

> * для обозначения цитат в языке Markdown используется знак "больше" [>]
>
> * его можно вставлять как перед каждой строкой цитаты, так и только перед первой строкой параграфа
>
>> * также синтаксис Markdown позволяет создавать вложенные цитаты (цитаты внутри цитат), используя двойной знак "больше" [>>]

## Работа с таблицами

Синтаксис языка Markdown позволяет создавать таблицы. Для этого используется символ вертикальной черты [|] для разделения ячеек и дефисы [-] для создания строки заголовка

| Column 1 | Column 2 | Column 3 |
|----------|----------|----------|
| Row 1    | Cell 2   | Cell 3   |
| Row 2    | Cell 5   | Cell 6   |
| Row 3    | Cell 8   | Cell 9   |

Управлять выравниванием текста модно при помощи двоеточий [:] по разные сторны от строки с тире.

| Column 1 | Column 2 | Column 3 |
|:---------|:--------:|---------:|
| Row 1    | Cell 2   | Cell 3   |
| Row 2    | Cell 5   | Cell 6   |
| Row 3    | Cell 8   | Cell 9   |

## Конфликты слияния в git
Слияние и конфликты являются неотъемлемой частью работы с Git.
Git позволяет выполнять слияния очень просто. В большинстве случаев Git самостоятельно решает, как автоматически интегрировать новые изменения.

Обычно конфликты возникают, когда два человека изменяют одни и те же строки в файле или один разработчик удаляет файл, который в это время изменяет другой разработчик. В таких случаях Git не может автоматически определить, какое изменение является правильным.

Git помечает файл как конфликтующий и останавливает процесс слияния. В этом случае ответственность за разрешение конфликта несут разработчики.

Системы контроля версий предназначены для управления дополнениями, вносимыми в проект множеством распределенных авторов (обычно разработчиков). Иногда один и тот же контент могут редактировать сразу несколько разработчиков. Если разработчик A попытается изменить код, который редактирует разработчик B, может произойти конфликт. Для предотвращения конфликтов разработчики работают в отдельных изолированных ветках. Основная задача команды git merge заключается в слиянии отдельных веток и разрешении любых конфликтующих правок.

![Конфликт](Конфликт.jpg)

При возникновении конфликта слияния git укажит о конфликте и предложит варианты решения: 

- отавить актуальную версию

- выполнить замену

- объединить оба варианта

А затем можно в ручную отредактировать необходимые тонкости.

## Работа в git с удаленным репозиторием

**Удаленный (иногда говорят "внешний") репозиторий** – это версии вашего проекта, сохраненные на удаленном сервере. Доступ к репозиторию на таком сервере может осуществляться по интернету или по локальной сети.

![Схема](Удаленный_%20репозиторий.jpg)

Основные этапы:

1. Создать аккаунт на Github.com

2. Создать локальный репозиторий

3. "Подружить" локальный и удаленный репозиторий. Github при создании нового репозитория подскажет как это можно сделать

4. Отправить (push) локальный репозиторий в удаленный на Github, при этом, возможно, понадобится авторизация

5. Провести изменения "с другого компьютера"

6. Выукачать (pull) актуальное состояние из удаленного репозитория

## Как сделать pull reques
 1. Сделать **Fork** репозитория

 2. Сделать **Clone** своей версии репозитория

 3. Создать ветку и внести в нее свои изменения

 4. Фиксируем изменения (делаем коммиты)

 5. Отпарвить свою версию всвой GitHub

6. На GitHub нажиаем кнопку **Pull request**
